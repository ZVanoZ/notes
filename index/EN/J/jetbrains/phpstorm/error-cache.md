# Описание

Приложение внезапно упало. 
Попытался перезапустить, но оно все равно падает.

## Диагностика

```shell
sudo journalctl --since "30 min ago" | grep -i phpstorm`
```

1. Первое падение (19:25:41)
```text
Process 32547 (phpstorm) dumped core.
Module .../tmp/sqlite-3.50.3.0-...-libsqlitejdbc.so without build-id.
```

2. Попытка перезапуска и предупреждение (19:26:13)

Вы запустили программу снова, и она попыталась восстановиться:
```text
WARN - #c.i.o.v.n.p.FSRecords - Records storage [.../records.dat] was in use by process ... which is not exist now
WARN - #c.i.o.v.n.p.m.c.VFSContentStorageOverMMappedFile - Content map[...] is empty ... re-building map from the storage
```
PhpStorm увидел, что предыдущая сессия «умерла» некорректно (файл блокировки records.dat остался активным). Он попытался принудительно перехватить права на файлы и начал перестраивать карту виртуальной файловой системы (VFS) из поврежденного хранилища.

3. Второе падение (19:27:41)
```text
Process 1032279 (phpstorm) dumped core.
Module .../libsqlitejdbc.so without build-id.
```
Спустя полторы минуты сканирования поврежденного кэша, Java-машина снова наступила на те же «грабли» (битые данные в SQLite кэше) и повторно рухнула в Segmentation Fault (то самое сообщение Consumed 5min 7.809s CPU time, с которого начался наш прошлый диалог).


## **Как это исправить?**

Пока вы полностью не удалите поврежденный индекс, PhpStorm так и будет циклически падать при каждой попытке прочесть этот файл. Нам нужно сбросить кэш «руками» через терминал.

Выполните следующие команды (убедитесь, что PhpStorm закрыт):
Шаг 1: Очищаем кэш конкретной версии

Удаляем временные файлы SQLite и индекс VFS, которые вызывают сбой:
```shell
rm -rf /home/ivan/.cache/JetBrains/PhpStorm2025.3/caches
rm -rf /home/ivan/.cache/JetBrains/PhpStorm2025.3/tmp
```

Шаг 2: Проверяем, где лежит проект

Если проект, который вы открывали, лежит на той самой подозрительной флешке — перенесите его на основной SSD/HDD компьютера. Если флешка во время индексации «отваливается» хотя бы на миллисекунду, SQLite база мгновенно ломается, что и приводит к libsqlitejdbc.so coredump.
Шаг 3: Запуск

Запустите PhpStorm. Он начнет индексацию проекта с абсолютно чистого листа (это займет чуть больше времени, чем обычно), но падать уже не должен.

Если после удаления папки caches и tmp падение повторится — дайте знать, посмотрим, не повреждена ли сама системная Java-машина (libjvm.so).
