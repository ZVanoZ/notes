# Что такое «Memory Bank» для Cursor?

**Это концепция.**  
Концепция пришла из другого ИИ-редактора — Cline (ранее Claude Dev), 
где разработчики и пользователи придумали систему из нескольких .md файлов для
обхода ограничений контекстного окна.

1. Разработчики Cursor создали механизм "Rules and Commands", который используется сообществом для реализации «Memory Bank» .
<pre>
На текущий момент [2026-01-13] разработчики Cursor (компания Anysphere) 
не выпускали официального шаблона.
Они сделали механизм «Memory Bank», но конфигурировать его нужно самому.

* .cursorrules или папка .cursor/rules/*.mdc: 
Официальный способ задавать инструкции и контекст.

* Codebase Indexing: 
Автоматическая индексация всего проекта, чтобы ИИ «знал» структуру без ручного ведения файлов.

* Notepads: 
Внутренняя фича Cursor для сохранения важного контекста.
</pre>
2. Сообщество создало несколько вариантов шаблонов "cursor-memory-bank", 
которые базируются на механизме "Rules and Commands".

## Итог:

<pre>
* Если вы хотите «официальный» путь — используйте документацию Cursor по Custom Rules. 
* Если вы хотите продвинутую систему «внешней памяти», чтобы ИИ не терял нить 
проекта на дистанции в несколько недель — репозиторий vanzan01 будет отличным 
выбором, хоть он и создан энтузиастом.
</pre>

## Концепция

* Зачем это надо?
  ```text
  Программа Cursor использует для работы различные LLM.
  При переключении между LLM происходит потеря контекста.
  В результате, генерируется противоречивый код и проект превращается в какашку.
  ```

* Как исправить ситуацию?  
**Использовать "Cursor Memory Bank", который подтянет контекст перед каждым обращением к LLM.**

* Как это работает?
  * Кратко: Создается набор *.md файлов, в которых описывается контекст.  
  * Средне:
  ```text
  1. Есть набор служебных *.md файлов 
  (Создаются самостоятельно; тянутся с github "vanzan01/cursor-memory-bank" либо из "ipenywis/aimemory", либо из других реализаций).
  Обычно расположены в субдиректориях "./cursor/..."
  В них описывается что делать программе Cursor при разных сценариях работы.
  Например: "Перед каждым обращением к LLM загляни в папку [memory-bank] и прочитай что делать".  
  2. Есть набор *.md файлов для проекта(создаются пользователем).
  Расположение зависит от набора служебных файлов из п.1. 
  В них описано что за проект, как он устроен, что нужно сделать.
  Например: "Проект для проксирования DNS-запросов. Базируется на NodeJS ^20. Язык TypeScript."
  ```

* Алгоритм работы

В Cursor есть два основных режима работы
1. Plan Mode - составление плана работы
2. Act Mode - выполнение плана, либо отдельного действия
