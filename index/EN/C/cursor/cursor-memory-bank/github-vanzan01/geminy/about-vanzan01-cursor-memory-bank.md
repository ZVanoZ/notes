# О системе vanzan01/cursor-memory-bank

## 1. Что такое vanzan01/cursor-memory-bank

**vanzan01/cursor-memory-bank** — это специализированный фреймворк контекстной памяти для ИИ-ассистентов (преимущественно Cursor), основанный на методологии «Memory Bank». Он представляет собой структурированный набор Markdown-файлов, которые служат внешним «хранилищем знаний» для ИИ внутри конкретного проекта.

Система позволяет LLM (Large Language Models) преодолевать ограничения контекстного окна, сохраняя долгосрочную память о принятых решениях, архитектуре, текущем прогрессе и целях проекта.

* **Репозиторий проекта:** [github.com/vanzan01/cursor-memory-bank](https://github.com/vanzan01/cursor-memory-bank)
* **Документация:** Подробные инструкции по внедрению находятся в файле `README.md` и `systemPrompt.md` внутри репозитория.

---

## 2. Назначение системы

Основная цель **cursor-memory-bank** — обеспечить «непрерывность сознания» ИИ-ассистента. Она решает следующие задачи:

* **Сохранение контекста:** ИИ не забывает архитектурные решения, принятые 100 сообщений назад.
* **Ускорение онбординга:** Позволяет ассистенту мгновенно включиться в работу над старым или сложным проектом.
* **Синхронизация целей:** Гарантирует, что ИИ понимает текущий этап работы и не предлагает решений, противоречащих стратегии развития проекта.
* **Документирование на лету:** Автоматизирует процесс ведения актуальной технической документации в процессе кодинга.

---

## 3. Структура системы

Система состоит из набора ключевых файлов, расположенных в директории `.cursor/memory-bank/` (или корне проекта), каждый из которых отвечает за свой аспект памяти:

1. **`projectbrief.md`**: Фундамент. Описание того, «зачем» и «что» мы строим. Цели и основные требования.
2. **`productContext.md`**: Описание функционала, пользовательского опыта и того, как проект должен работать с точки зрения продукта.
3. **`systemPatterns.md`**: Техническая ДНК. Архитектурные паттерны, структура директорий, выбранные технологии и стандарты кодирования.
4. **`techContext.md`**: Техническая среда. Зависимости, версии библиотек, особенности развертывания и технические ограничения.
5. **`activeContext.md`**: Текущее состояние. Над чем мы работаем прямо сейчас, какие изменения были внесены последними.
6. **`progress.md`**: Чек-лист. Что сделано, что осталось, и какие проблемы (known issues) существуют.

## 4. Когда использовать, а когда — нет

### ✅ Использовать, если:

* **Сложный проект:** Вы работаете над долгосрочным проектом с множеством взаимосвязей.
* **Частые перерывы:** Вы возвращаетесь к коду спустя время и не хотите заново объяснять ИИ структуру проекта.
* **Строгие стандарты:** Вам важно, чтобы ИИ следовал специфическим архитектурным паттернам (например, Clean Architecture или специфический стиль именования).
* **Ограничение контекста:** Вы замечаете, что ИИ начинает «галлюцинировать» или забывать важные детали проекта из-за большого объема файлов.

### ❌ Не использовать, если:

* **Простые скрипты:** Для написания одноразового файла на 50 строк накладные расходы на поддержание памяти избыточны.
* **Прототипирование («на коленке»):** Когда архитектура меняется каждые 5 минут, поддержание Memory Bank в актуальном состоянии может замедлить процесс.
* **Строгие ограничения безопасности:** Если политика компании запрещает хранение подробных архитектурных данных в текстовых файлах, к которым имеет доступ ИИ (хотя файлы хранятся локально).
